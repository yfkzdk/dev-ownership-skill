# BDD 响应协议 — 模块卡住诊断与治疗

> 对应测量金字塔 BDD 触发条件。
> 当前系统"只测不治"——BDD 触发了但没有动作。
> 本协议定义触发后的完整响应链。
> 开发者设计。10 步链条，不可跳跃。

---

## 触发条件（三选一）

- 一个功能模块的 commit 无法完成对应需求
- 模块卡住超过阈值时间（尝试多个方案后仍不 work）
- 测试绿了但实际使用中模块不 work（假通过）

## 响应链

```
① 发现问题 ──→ ② BDD触发 ──→ ③ 诊断 ──→ ④ 问题分类
                                              ↓
⑧ 重试 ←── ⑦ 对接问题 ←── ⑥ 搜索信息 ←── ⑤ 问题拆解
  ↓
⑨ 扫描过度设计 ──→ ⑩ 切忌造轮子
```

---

## ① 发现问题

开发者在开发过程中遇到障碍。信号包括：
- 同一个模块反复修改 3 次以上
- 测试始终过不了，但说不清为什么
- 功能在本地手动测试时表现异常
- 开发者在对话中表达困惑或卡壳

## ② BDD 触发

满足任一触发条件 → BDD 事件触发 → 进入诊断。
**此时不跳过、不推迟、不让 AI 接管。**

## ③ 诊断

AI 依次问开发者 3 个问题，必须全部回答：

**A1: 卡在哪个层面？**
- Spec 不明确（不知道"正确"是什么样）
- 技术实现（知道要什么，但写不出来）
- 测试（测试写了但断言过不了，或不确定测试对不对）

**A2: 已经试了什么？**
- 列出尝试过的方案（≥1 个）
- 每个方案为什么失败（具体错误信息/行为表现）

**A3: 是否有信息盲区？**
- 这个问题的底层机制你不确定是什么？
- 有没有你还没搜过的方面？

## ④ 问题分类

| 类别 | 判定标准 | 下一步 |
|------|---------|--------|
| Spec 问题 | spec 条款有歧义、不完整或前后矛盾 | 回到 Spec 阶段，仅重审该模块 |
| 技术实现问题 | spec 清楚，但不知道怎么写 | 进入 ⑤ 问题拆解 |
| 测试问题 | 测试结果与预期不符，或不确认测试是否写对了 | 先区分测试写错 vs 实现有误 |
| 依赖问题 | 外部 API/库/环境导致 | 进入 ⑤ 问题拆解（搜替代方案或 mock 策略） |

## ⑤ 问题拆解

**把一个卡住的大问题拆成 2-4 个子问题。**

原则：
- 每个子问题必须是**可独立搜索的**——有明确的关键词
- 子问题之间不重叠
- 如果拆不出 ≥2 个子问题 → 说明问题定义还不够清晰，回到 ③ 重新诊断

例子：
```
大问题: "状态机转换逻辑不对"

拆解:
  子问题1: "状态机模式 Python 实现方式有哪些" 
  子问题2: "采购订单状态转换的边界条件（什么情况下不能从A到B）"
  子问题3: "状态机测试——怎么验证状态转换正确性"
```

## ⑥ 搜索信息

**对每个子问题独立搜索，≥3 个信息源。**

搜索源优先级：
1. GitHub 开源项目（相似模块的实现参考）
2. 技术文档/官方 spec（Python docs、RFC、ISO 标准）
3. 行业方案（Stack Overflow、技术博客）——作为补充，不是主源

搜索规则（引用 research-protocol.md）：
- 每个子问题至少搜索一次
- 搜索结果必须有 URL 或明确出处
- AI 总结搜索结果，标注：发现了什么 / 不适用什么 / 不确定什么

**此步骤禁止跳过。禁止 AI 凭内部知识直接给方案。**

## ⑦ 对接问题

**将搜索结果映射回原始问题。**

AI 必须回答：
- 搜索结果中**哪个方案/代码片段**可能解决当前子问题？
- 这个方案需要**做什么调整**才能适配当前项目？
- 有没有搜索结果**明确不适用**的？（可以少走弯路）
- 整个搜索过程中，有没有发现**已有现成的库/工具可以直接用**？（→ ⑩ 切忌造轮子）

## ⑧ 重试

基于 ⑥ 搜索 + ⑦ 对接的结果，重新实现。

```
重试次数: 最多 2 轮
每轮: 修改 → 测试 → 开发者确认（不是 AI 确认）
2 轮后仍卡住 → 升级（⑩ 之后进入升级）
```

重试期间限制：
- Feynman 门禁不可跳过（这个模块已有债务）
- 不允许 AI 跳过搜索直接生成方案
- 该模块 CDR 暂时冻结，不计入当前统计

## ⑨ 扫描过度设计

重试成功后，**对实现进行过度设计扫描**。

检查项：
- 这个方案是否引入了不必要的抽象层？
- 代码行数是否显著超过 spec 要求的复杂度？
- 是否用了 spec 未要求的库/依赖？
- 方案中是否有"未来可能需要"的功能？→ **删除，等需要时再加**
- 是否实现了 spec 范围外的功能？→ **如果 spec 未要求，删除**

引用硬规则第 9 条（R14）：每项设计变更必须逐条对应到 spec 条款。无 spec 依据的代码 → 删除。

## ⑩ 切忌造轮子

最终检查：

- 搜索结果中是否有**现成的库/工具**可以实现同样的功能？
- 如果有，为什么没用？（必须有明确理由，写入 ADR 补充）
- 如果没有现成的，是否有**参考实现**可以大幅简化自己的方案？

## 升级（重试 2 轮后仍卡住）

### 升级类型

**设计级升级**:
```
触发: 技术实现多次失败
动作:
  - 标记模块 "设计风险"
  - 强制 Feynman 门禁（不可跳过）
  - 门禁配额不消耗（设计问题，非开发者逃避）
  - Retrospect 单独审视此模块
```

**范围级升级**:
```
触发: Spec 修订 ≥2 次仍卡住
动作:
  - 暂停此模块
  - 标记 spec 条款 "待定"
  - 先完成其他模块
  - 如果是核心路径 → 重新评估项目范围
```

**能力级升级**:
```
触发: 开发者无法在 Feynman 门禁中解释卡住原因
动作:
  - 标记知识领域 "待学习"
  - 推荐具体学习资源（文档/论文/规范章节）
  - 下个项目自评量表追加此领域追问
  - 不强迫——能力缺口不是错
```

## BDD 事件记录格式

```yaml
bdd_event:
  timestamp: "2026-06-16T14:30:00"
  module: "purchase_order_state_machine"
  trigger: "implementation_stuck"
  diagnosis: "状态转换边界条件不明确"
  classification: "技术实现"
  decomposition:
    - "状态机模式 Python 实现方式"
    - "采购订单状态转换的边界条件"
  sources:
    - "https://github.com/... (状态机参考实现)"
    - "https://docs.python.org/... (enum 文档)"
  solution_found: true
  over_engineering_scan: "pass"
  reinvent_wheel_check: "pass — 无现成库，参考实现已适配"
  rounds: 1
  resolution: "resolved"
```

## 与现有系统集成

| 模块 | 集成点 |
|------|--------|
| research-protocol | ⑥ 搜索——引用搜索规范 |
| hard-rule #5 | ⑥ 搜索——设计方案前必须搜索 |
| hard-rule #9 (R14) | ⑨ 扫描——无 spec 依据的代码删除 |
| gate-quota-tracker | 升级时不消耗配额 |
| cdr-sr-tracker | 卡住期间 CDR 冻结 |
| Feynman gate | 升级时强制触发，不可跳过 |
| session-log v2 | BDD 事件单独记录 |
| retrospect | 所有升级模块必须单独审视 |
