# BDD 响应协议 — 模块卡住诊断与治疗

> 对应测量金字塔 BDD 触发条件。
> 当前系统"只测不治"——BDD 触发了但没有动作。
> 本协议定义触发后的完整响应链。

---

## 触发条件（三选一）

- 一个功能模块的 commit 无法完成对应需求
- 模块卡住超过阈值时间（尝试多个方案后仍不 work）
- 测试绿了但实际使用中模块不 work（假通过）

## 响应链

```
BDD触发
  ↓
Phase A: 诊断（问 3 个问题，5 分钟内完成）
  ↓
Phase B: 分类处理（根据诊断结果分支）
  ↓
Phase C: 重新尝试（限时）
  ↓
Phase D: 升级（如果仍卡住）
```

---

## Phase A: 诊断

AI 必须按顺序问开发者 3 个问题，不允许跳过：

**A1: 卡在哪个层面？**
- Spec 不明确？（不知道"正确"是什么样）
- 技术实现？（知道要什么，但写不出来）
- 测试？（测试写了，但断言过不了，或不确认测试对不对）
- 其他？（外部依赖、环境问题、数据问题）

**A2: 已经试了什么？**
- 开发者列出尝试过的方案（≥1 个）
- 每个方案为什么失败（具体错误信息/行为表现）

**A3: 这个模块的 spec 需要修改吗？**
- 如果 spec 没问题，是实现的锅 → 进入 Phase B 实现分支
- 如果 spec 本身就有歧义 → 回到 Spec 阶段，只重审这个模块的 spec 条款

---

## Phase B: 分类处理

### B1: Spec 问题

```
→ 回到 Spec 阶段（仅此模块）
→ 重新定义这个模块的：前置条件 / 后置条件 / 边界
→ 修订后的 spec 条款标记为 "revision:1"
→ revision count 计入 spec 质量追踪
→ Feynman 门禁：开发者必须解释"之前的 spec 哪里不明确，现在为什么明确了"
→ 重新进入 TDD：为修订后的 spec 写新测试
```

### B2: 技术实现问题

```
→ AI 提出 2-3 个替换方案（不是 1 个——必须给选择）
→ 每个方案标注：复杂度 / 风险 / 是否已有类似实现可参考
→ 开发者选一个方案 → 写 ADR 补充（记录"为什么原方案失败，为什么选这个"）
→ 如果涉及架构变更 → 升级到设计级门禁（不可跳过）
→ 重新实现 → 限时 2 次尝试
```

### B3: 测试问题

```
先确认是哪个：
  B3a: 测试写错了 → 修正测试，参考 spec 条款逐条对齐
  B3b: 测试对了但实现有误 → 回到 B2
  B3c: "测试绿了但实际不 work"（假通过）
       → 这是最危险的情况——断言不够强
       → 检查：这个模块的突变得分是多少？
       → 如果 < 70% → 先增强断言，重新突变测试
       → 如果 ≥ 70% → 测试已经够了，问题可能在集成层，检查模块间交互
```

---

## Phase C: 重新尝试（限时）

```
重新尝试次数: 最多 2 轮
每轮包含: 修改 → 测试 → 开发者确认（不是 AI 自己确认）
2 轮后仍卡住 → 进 Phase D 升级
```

### 重新尝试期间的限制
- 不允许跳过 Feynman 门禁（这个模块已经卡住了，跳过只会堆积债务）
- 不允许用 AI 生成替代方案然后不经审查直接 Accept
- 该模块的 CDR 暂时冻结——新 AI 生成代码自动标记为"待验证"，不计入当前 CDR

---

## Phase D: 升级

### D1: 设计级升级
```
触发条件: B2 (技术实现问题) 且重试 2 轮仍卡住
动作:
  - 标记此模块为 "设计风险"
  - 强制 Feynman 门禁（不可跳过）: 开发者必须解释"问题的根本原因是什么设计缺陷"
  - 这个模块的门禁配额不消耗（不是开发者的错，是设计的问题）
  - 进入 Retrospect 时必须单独审视这个模块
```

### D2: 范围升级
```
触发条件: B1 (spec 问题) 且修订 2 次后仍卡住
动作:
  - 暂停此模块的开发
  - 标记 spec 条款为 "待定"
  - 先完成其他模块，最后回到这个模块
  - 如果这个模块是核心路径上的: 重新评估项目范围——是否应该缩小
```

### D3: 能力升级
```
触发条件: 开发者在 Feynman 门禁中无法解释卡住的原因
动作:
  - 不强迫
  - 标记此模块的知识领域为 "待学习"
  - 推荐学习资源（具体的技术文档/论文/规范章节，不是泛泛的"学一下 X"）
  - 下一个项目启动前，自评量表中追问此领域
```

## BDD 记录格式

每次 BDD 触发，在 session-log v2 中记录：

```yaml
bdd_event:
  timestamp: "2026-06-16T14:30:00"
  module: "purchase_order_state_machine"
  trigger: "implementation_stuck"  # spec_ambiguous | implementation_stuck | test_false_pass | test_failure
  diagnosis: "技术实现——状态转换的边界条件不明确"
  attempted: ["方案A: if-else链", "方案B: 状态模式"]
  branch: "B2"
  rounds: 2
  resolution: "escalated_to_D1"  # resolved | escalated_to_D1 | escalated_to_D2 | escalated_to_D3
  spec_revision_count: 0
```

## 与现有系统的集成

| 现有模块 | 集成点 |
|---------|--------|
| gate-quota-tracker | B2/B3 升级时**不消耗**配额（这是设计问题，不是开发者逃避） |
| cdr-sr-tracker | 卡住期间该模块 AI 代码自动标记，不计入当前 CDR |
| Feynman gate | D1/D2/D3 升级时**强制触发**，不可跳过 |
| session-log v2 | BDD 事件单独记录（格式如上） |
| retrospect | 所有 D1/D2/D3 升级的模块必须单独审视 |
