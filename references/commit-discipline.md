# 结构化提交规范

> [来源: Conventional Commits v1.0.0 — conventionalcommits.org]
> [来源: 0x404/ccs_dataset — 88,704条commit, 116个开源项目的真实类型分布]
> [来源: Git Documentation/SubmittingPatches — "Make separate commits for logically separate changes"]
> [来源: Linux kernel patch submission guidelines — each commit builds and runs properly]
> [来源: TituxMetal#19 + cc-foundry + nimblehq#604 — 选择性暂存/质量门/8步pipeline]
> [来源: workflow-patterns — TDD 11-step lifecycle with atomic commits]
> [来源: backend-atomic-commit (diversioteam) — atomicity check via staged diff coherence]

## 硬性规则

1. **一个阶段至少一个commit。** 禁止跨阶段合并。禁止事后整理commit历史。
2. **TDD每个子阶段独立commit。** RED(test)→GREEN(feat/fix)→REFACTOR(refactor) 不允许合并为一个 `feat`。
3. **禁止 `git add -A` 和 `git add .`。** 只暂存与当前变更逻辑相关的文件。每 commit 必须显式指定文件路径。
4. **依赖顺序提交。** 阶段内部: 基础设施 → 核心逻辑 → 测试 → 文档。
5. **文档与代码分开提交。** DESIGN.md 和 src/ 不在同一个commit。
6. **Commit message必须包含spec条款引用。**
7. **同一feature的commit之间必须通过Refs字段关联。**

## 原子性规则（来源: Git SubmittingPatches + Linux kernel）

8. **每个 commit 必须是一个逻辑变更。** 如果 commit message 需要用"和"字来描述 → 拆成两个 commit。
   ```
   合法: "feat(server): add select()-based event loop"
   合法: "fix(parser): handle null bulk string $-1 in RESP decoder"
   不合法: "feat: add server AND fix parser bug AND update README" → 拆成 3 个 commit
   ```

9. **每个 commit 之后项目必须处于可构建+测试全绿状态。** 这是 Linux 内核的硬性标准——保证 `git bisect` 能在任何一个 commit 上正常工作。如果 TDD RED 阶段的 commit 测试会失败——使用 `--no-verify` 并在 body 中记录原因。

10. **代码移动和代码修改分两个 commit。** 先从 A 文件移到 B 文件（纯移动，无修改）。再在 B 文件中修改。这样 diff 清晰——reviewer 看到第一个 commit 是纯 copy-paste，第二个 commit 才是实质性改动。

11. **格式化/清理和逻辑修改分两个 commit。** ruff fix → 独立 commit (`style:`)。逻辑修改 → 独立 commit (`feat:` 或 `fix:`)。不混在一起。

## 阶段特定规则

12. **Review 和 Retrospect 必须是独立的 commit**——禁止合并为 `docs(review,retrospect)`。Review 审查当前代码。Retrospect 反思整个过程。不同的活动 → 不同的 commit。

13. **Commit message 必须由开发者撰写。** 禁止 AI 签名(`Co-Authored-By: Claude`)。禁止 AI 生成标记(`Generated by`、`via Claude Code`)。AI 辅助的事实记录在 session log 中。Commit message 只记录**开发者做的设计决策**。

## Commit Type 定义

> 来源: Conventional Commits v1.0.0 + 0x404 研究团队的精准定义（减少类型重叠）

| Type | 用途 | 判断标准 | 真实占比 |
|------|------|---------|:--:|
| `fix` | 修复 bug | 修复了与 spec 不符的行为 | 27.0% |
| `chore` | 维护任务 | 不修改 src 或 test 文件 | 26.0% |
| `feat` | 新功能 | 新增了 spec 中定义的功能 | 17.2% |
| `refactor` | 重构 | 改结构不改行为，测试全绿 | 9.6% |
| `docs` | 文档 | 只改 .md / 注释 | 8.1% |
| `test` | 测试 | 只添加/修改测试，不改源码 | 6.0% |
| `build` | 构建系统 | 改 pyproject.toml / 依赖 / 打包 | 3.5% |
| `ci` | CI/CD | 改 .github/workflows / 流水线 | 1.3% |
| `style` | 代码风格 | 只改格式/空格/lint fix (无逻辑变化) | 0.9% |
| `perf` | 性能 | 改进性能，不改行为 | 0.4% |

## Type 选择决策树

```
改了什么？
├── 只改了 .md 文件？                                    → docs
├── 只改了测试文件(tests/)？                              → test
├── 只改了 CI/构建配置？                                  → ci / build
├── 只改了格式(空格/换行/ruff fix)？                      → style
├── 修了一个与 spec 不一致的行为？                         → fix
├── 新增了 spec 中定义的功能？                             → feat
├── 改进已有代码结构，测试全绿，行为不变？                  → refactor
├── 改进性能，行为不变？                                   → perf
└── 都不符合？                                            → chore
```

## 阶段内 Commit 序列

### Spec阶段
```
docs(spec): add formal specification for <feature> [spec-§1-§N]
```

### Design阶段
```
docs(design): add architecture design with ADRs [spec-§1-§N]
```

### TDD阶段 — 必须拆分为三个独立commit

```
RED:
  test(<scope>): add failing tests for <feature> [spec-§X.Y]

GREEN:
  feat(<scope>): implement <feature> [spec-§X.Y]
  Refs: <test-commit-hash>
  (或 fix(<scope>): 如果是修bug)

REFACTOR (如有):
  refactor(<scope>): <what was restructured> [spec-§X.Y]
  Refs: <impl-commit-hash>
```

### Review阶段
```
fix(<scope>): <具体修复内容> [spec-§X.Y]
(多个fix分开commit——一个问题一个commit)
```

### Retrospect阶段
```
docs(retrospect): project review with traceability audit
```

## 提交前自检

每个commit之前，AI执行：

1. [ ] 这个 commit 能用一句话描述吗（不带"和"字）？（Rule 8）
2. [ ] 这个 commit 之后项目能构建+测试全绿吗？（Rule 9）
3. [ ] Staged files中是否有代码移动和修改混合？（Rule 10）
4. [ ] Staged files中是否有格式化和逻辑修改混合？（Rule 11）
5. [ ] Commit message 中是否禁用了 AI 签名？（Rule 13）
6. [ ] 是否使用了 `git add -A` 或 `git add .`？（如果是→阻断）
7. [ ] Commit message 是否包含 spec 条款引用？
8. [ ] 如果是系列commit，Refs 字段是否指向了前一个commit？

## `--no-verify` 使用规则

1. **每次使用 `--no-verify` 必须在 commit body 中记录原因**：
   ```
   --no-verify: <具体原因>
   例: --no-verify: TDD RED phase — tests expected to fail (Rule 9例外)
   例: --no-verify: coverage gap (__main__.py 0%) — deferred to next iteration
   ```

2. **禁止原因**（出现即违规）:
   - "为了快速提交"
   - "lint 问题太多懒得修"
   - 空白或无原因

3. **Retrospect 统计**: 项目结束时，AI 必须统计 `--no-verify` 次数和原因分布。

## 反模式

| 反模式 | 为什么错 | 来源 |
|--------|---------|------|
| 一个 commit 做多件事(用"和"才能描述) | 违反"一个逻辑变更"原则——Git 官方标准 | Rule 8 |
| commit 之后代码无法构建 | 破坏 git bisect——Linux 内核硬性标准 | Rule 9 |
| 代码移动+修改混在一起 | diff 无法区分"移动了什么"和"改了什么" | Rule 10 |
| 格式化+逻辑混在一起 | reviewer 在空白行里找逻辑变更 | Rule 11 |
| Review+Retrospect 合并为 `docs(review,retrospect)` | 不同的活动 → 不同的 commit | Rule 12 |
| `Co-Authored-By: Claude` | commit message 是开发者写的——不是 AI | Rule 13 |

## 违规检测（6 项自动检测）

```
检测 1: 逻辑独立性 (Rule 8)
  信号: commit message 含 "and"/"和"/"同时"/"以及"
        OR staged files 跨越 3+ 不相关模块
        OR diff 混合了新功能+bug修复+格式化
  处置: [BLOCKING] 拆分成独立 commit

检测 2: 可构建性 (Rule 9)
  如果 staged files 含源文件:
    运行测试 → 失败 → [BLOCKING]（TDD RED 阶段除外, 需 --no-verify）
    运行测试 → 通过 → 放行

检测 3: 代码移动与修改混合 (Rule 10)
  如果同一段代码同时出现在删除行(旧文件)和新增行(新文件),
  且内容有差异(不是纯移动):
    [WARN] 先做纯移动 commit，再做修改 commit

检测 4: 格式化与逻辑混合 (Rule 11)
  如果 diff 同时含纯格式变更(空白/缩进/import排序)
  AND 逻辑变更(新函数/条件/变量):
    [BLOCKING] 格式化和逻辑修改必须分两个 commit

检测 5: Review/Retrospect 合并 (Rule 12)
  如果 commit message 同时含 "review" 和 "retrospect":
    [BLOCKING] Review 和 Retrospect 必须独立 commit

检测 6: AI 签名 (Rule 13)
  如果 commit message 含 "Co-Authored-By: Claude|OpenAI"
  OR "Generated by" OR "via Claude Code" OR "AI-assisted":
    [BLOCKING] AI 辅助记录在 session log, 不在 commit message
```

## 示例: 一个完整的 TDD commit 序列

```
test(order): add failing test for line item total calculation  [spec-§4.1]
feat(order): implement line item total calculation  [spec-§4.1]
  Refs: a1b2c3d
refactor(order): extract calculate_line_total private method  [spec-§4.1]
  Refs: e4f5g6h
```

每个 commit 独立可 revert，测试全绿，意图清晰。
