# Feynman Gate 规范

> [来源: koukekoukej-glitch/feynman-tutor — "makes YOU teach the AI to expose gaps in your understanding"]
> [来源: Kazemitabaar et al. 2024 — Lead-and-Reveal technique: 开发者在代码揭示前必须解释下一步]
> [来源: Kilo Learn — embedded comprehension checks with real code identifiers after generation]

## 原理

每个阶段出口设置门禁。**Commit保留工作记录，Feynman门禁检查开发者是否真正理解。**
门禁不通过 → commit不撤销，但必须回到该阶段针对性补足 → 重新过门禁。

## 执行顺序

```
阶段执行完成 → AI自检 → Commit → Feynman门禁 → 通过/不通过
                                          │
                                    不通过 → 补足 → 重新门禁
```

## 门禁问题

每个阶段结束时，AI基于**该阶段真实产出物**提出2-3个问题。
问题必须引用实际代码标识符（函数名/变量名/类名）或实际设计条款——不使用泛泛的抽象问题。

### Spec阶段出口

1. "这个功能解决谁的什么问题？用一句话描述"
2. "不在范围内的最大一项是什么？为什么排除？"
3. "在 SPEC 的验收标准中，指出一条最容易在实现中被忽略的，解释为什么容易被忽略"

### Design阶段出口

1. "在 ADR 中，你拒绝了哪个备选方案？引用 ADR 的具体条款号，解释拒绝的真正原因"
2. "当前方案最大的风险点对应 DESIGN 中哪一节？如果出问题会是什么症状？"
3. "DESIGN 中是否有 SPEC 未要求的额外设计？如果有，对应的 SPEC 条款是什么？如果没有，你确认过吗？"

### TDD阶段出口

1. "在 `[测试文件名]:[行号]` 中，参数 `[具体参数名]` 为什么选择 `[具体值]` 而不是其他值？换一个值会导致什么结果？"
2. "`[源文件名]:[行号]` 的 `[函数名]` 的边界情况有哪些？你的测试 `[测试函数名]` 覆盖了哪几个？"
3. "如果 `[源文件名]:[行号]` 这行有一个单字符错误（比如 `+` 写成 `-`，`>` 写成 `>=`），你的测试断言 `[测试函数名]` 能抓到吗？"

### Review阶段出口

1. "你在5轴审查中发现的3个最大风险是什么？引用具体文件:行号，从高到低排"
2. "有没有代码你接受了但没有完全理解？指出具体文件:行号和让你困惑的表达式"
3. "`[源文件名]` 的 `[类名/函数名]` 如果用完全不同的方案重写，哪个方案？变化最大的是什么？"

### Retrospect阶段出口

1. "这次开发中你犯了什么错误？根本原因是什么？引用具体文件:行号"
2. "下次类似项目，你会哪里做不同？具体到流程/工具/检查点"
3. "有什么新陷阱应该加入 known-traps.md？写下条目草稿（发现场景/症状/根因/修复）"

## 通过标准

- 开发者用**自己的话**表达，而非复述AI的总结或文档原文
- 能说出"**为什么**"，而非仅说出"是什么"
- 卡壳/沉默/翻代码超过30秒 → 不通过，该阶段需针对性补足
- 回答中出现"AI说的"、"文档写的"、"应该是这样吧" → 不通过

## 失败处理

1. AI指出具体缺陷，引用代码行号或设计条款号
2. 开发者回到该阶段针对性补足（不是重做整个阶段）
3. 补足后重新过门禁
4. 同一门禁最多重复3次。3次不通过 → 暂停，建议开发者手动学习后重试，不继续推进

## AI提问规范

- 问题必须来自当前阶段的**真实产出**，不能是模板填空
- 问题引用具体的函数名、变量名、类名、spec条款号
- 每次门禁至少1个问题是追问式的（在开发者回答了第一个问题后，基于其回答内容继续深挖）
- 示例：
  ```
  AI: "Money.__mul__ 方法里为什么要用 Decimal(str(scalar)) 而不是直接 Decimal(scalar)？"
  开发者: "为了防止float精度丢失"
  AI: "好。那如果scalar已经是Decimal类型，走这条路会不会有问题？str(Decimal('0.1')) 和直接传 Decimal('0.1') 有什么区别？"
  ```
