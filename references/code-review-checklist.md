# 五轴代码审查清单

> [来源: bouob/coding-skills — 5-axis review: design/readability/performance/security/testability]
> [来源: evanklem/evanflow — Five Failure Modes + assertion-correctness check]
> [来源: 面试问题诊断书 §2.4 — "审查能力依赖AI对照而非自身知识"]

## 使用规则

1. **开发者先用此清单逐项自查**，记录发现的问题
2. AI同时标记潜在问题
3. 双方对比结果，讨论差异点
4. AI的标记是辅助，开发者的判断是最终依据
5. **禁止跳过自查直接看AI的标记**

## 轴1: 设计 (Design)

检查代码结构是否合理。

| # | 检查项 | 怎么判断通过 |
|---|--------|-------------|
| 1.1 | 单一是职责 | 每个类/函数可以用一句话描述职责，且这句话里没有"和"字 |
| 1.2 | 依赖注入 | 外部依赖通过构造函数传入，不是内部new/import后直接调用 |
| 1.3 | 耦合方向 | 领域层不依赖基础设施层。检查import方向 |
| 1.4 | 过度设计 | 当前需求≤3条规则时不用策略模式。如果代码中出现了当前不需要的抽象层→标记 |

## 轴2: 可读性 (Readability)

检查代码是否容易理解。

| # | 检查项 | 怎么判断通过 |
|---|--------|-------------|
| 2.1 | 命名精准 | 变量名/函数名体现"做什么"而非"怎么做"。不用data/info/result/tmp |
| 2.2 | 函数长度 | 单个函数≤50行。超过→应拆分 |
| 2.3 | 嵌套深度 | 最深嵌套≤3层。超过→提取子函数或用guard clause |
| 2.4 | 魔法数字 | 所有非0/1/-1的字面量都有命名常量。`60`→`SECONDS_PER_MINUTE` |
| 2.5 | 注释质量 | 注释解释"为什么"，不是"做什么"（代码已经说了做什么） |

## 轴3: 性能 (Performance)

检查是否有明显的性能隐患。

| # | 检查项 | 怎么判断通过 |
|---|--------|-------------|
| 3.1 | N+1查询 | 循环内没有数据库调用或网络请求 |
| 3.2 | 重复计算 | 热路径上没有每次调用都重新排序/过滤的逻辑。`sorted()`结果应缓存 |
| 3.3 | 内存泄漏 | 集合/缓存有大小上限或过期机制 |

## 轴4: 安全 (Security)

检查是否有安全漏洞。

| # | 检查项 | 怎么判断通过 |
|---|--------|-------------|
| 4.1 | 注入防范 | SQL使用参数化查询，不使用字符串拼接 |
| 4.2 | 硬编码密钥 | 所有密钥/Token/密码从环境变量读取，不在代码中 |
| 4.3 | 输入校验 | 所有外部输入（API参数/文件内容/用户输入）在边界处校验 |
| 4.4 | 错误信息泄露 | 异常消息不暴露内部实现细节（文件路径/数据库结构/调用栈） |

## 轴5: 可测性 (Testability)

检查代码是否容易测试。

| # | 检查项 | 怎么判断通过 |
|---|--------|-------------|
| 5.1 | 全局状态 | 没有模块级别的可变全局变量 |
| 5.2 | 外部依赖可注入 | 需要mock的外部调用都通过接口/参数传入 |
| 5.3 | 副作用隔离 | 有副作用的代码（I/O/随机/时间）集中在可替换的模块中 |

## 断言正确性检查

> [来源: arXiv 2402.13521 — 62%+ of LLM-generated assertions fail one-character-bug check]

对每个测试断言，自问：
"如果实现代码中有一个单字符错误（比如 `>` 写成 `>=`，`+` 写成 `-`），这个断言会失败吗？"
如果答案是不会 → 这个断言不够强，需要补充。
