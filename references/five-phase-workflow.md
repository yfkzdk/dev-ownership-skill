# 五阶段详细交互点

> [来源: evanklem/evanflow — 每阶段checkpoint + 强制人类确认]
> [来源: BugRoger/beastmode — design→plan→implement→validate→release]

## 核心原则

每个阶段至少3个强制交互点，确保开发者全程参与，杜绝黑箱开发。

| 交互点 | 位置 | 谁主导 | 内容 |
|--------|------|--------|------|
| **入口确认** | 阶段开始 | AI解释 → 开发者确认 | AI说明本阶段目标、步骤、预期产出。开发者确认后开始 |
| **中间审查** | 阶段中途 | AI展示 → 开发者决策 | 至少一个中间产物需开发者做出设计决策（非Accept/Reject二元选择） |
| **出口门禁** | 阶段结束 | AI提问 → 开发者表达 | Feynman门禁：开发者用自己的话解释后方可进入下一阶段 |

## 各阶段交互时序

### 阶段1: Spec

```
AI: "本阶段目标是将需求转化为形式化规格。步骤：需求溯源四问 → 边界定义 → 前后置条件 → 测试场景推导"

[入口确认] AI简述理解的需求 → 开发者确认

[执行]
  AI: 需求溯源四问
  开发者: 逐一回答
  AI: 生成规格草稿 + AI推理摘要
  AI: "以下是规格草稿。请特别关注边界定义——如果边界错了，后面所有工作都偏了"

[中间审查] 
  AI展示边界定义 + 关键验收标准 → 
  开发者决策："边界是否可以？有没有遗漏的利益相关者场景？"

AI根据反馈修正 → 生成完整规格

[AI自检] AI输出自检报告（验收标准可测试性/边界清晰度/异常路径覆盖）

[Commit] 规格文档提交

[Feynman门禁] AI提问 → 开发者用自己的话回答 → AI判断通过/不通过
```

### 阶段2: Design

```
[入口确认] AI简述Spec要点 → 开发者确认理解一致

[前置搜索] 
  AI执行research-protocol.md定义的搜索流程
  搜索过程记录到session log

[执行]
  AI: 生成架构设计草稿（限界上下文/领域模型/ADR/备选方案对比）+ AI推理摘要

[中间审查]
  AI展示备选方案对比表 → 
  开发者决策："选哪个方案？为什么？"
  （这是设计阶段最重要的交互——开发者必须做架构决策，AI不能代劳）

AI根据决策生成完整设计文档

[AI自检] AI输出自检报告（决策理由完整性/备选方案充分性/是否过度设计）

[Commit] 设计文档提交

[Feynman门禁] AI提问 → 开发者用自己的话回答
```

### 阶段3: TDD

```
[入口确认] AI简述设计要点 → 开发者确认

[执行 - RED]
  AI: 根据Spec测试场景生成测试用例骨架（标注每个参数的选择理由）
  AI: "以下是测试用例设计。请审查：参数值是否合理？边界是否覆盖？"
  
[中间审查]
  开发者审查测试用例 → 
  开发者决策："哪些测试场景需要增删？参数值是否需要调整？"
  AI根据反馈修正

[执行 - GREEN + REFACTOR]
  AI: 生成最小实现使测试通过，附带逐行解释 + AI推理摘要
  AI: 运行测试，展示结果
  AI: 如有重构机会，展示before/after

[AI自检] AI输出自检报告（边界覆盖/断言有效性/双路径可用性）

[Commit] 测试commit → 实现commit（分开提交）

[Feynman门禁] AI提问 → 开发者用自己的话回答
```

### 阶段4: Review

```
[入口确认] AI简述当前代码状态 → 开发者确认

[执行]
  开发者: 使用5轴清单逐项自查
  AI: 同时标记潜在问题

[中间审查]
  开发者展示自查结果 → 
  AI展示标记的问题 →
  双方讨论差异点 → 
  开发者决策："哪些是真正的问题？优先级？"

AI根据决策生成修正方案

[需求一致性检查]
  AI: 逐条对照Spec验收标准，检查当前实现是否仍然对齐
  如有偏离: "以下实现与Spec §X.Y不一致：[具体偏离]。需要更新实现还是更新Spec？"
  开发者决策

[AI自检] AI输出自检报告（spec覆盖率/无主代码检测/修正建议分级）

[Commit] review修正commit

[Feynman门禁] AI提问 → 开发者用自己的话回答
```

### 阶段5: Retrospect

```
[入口确认] AI确认所有代码已合并 → 开发者确认

[Step 1: 可追溯性审计]
  AI: 逐类遍历，生成双向追溯矩阵 + 合规评分
  详见 traceability-audit.md

[中间审查]
  AI展示无主代码清单 + 实现缺失清单 →
  开发者对每个标记项做处置决策（删除/补spec/补实现）
  开发者决策："哪些需要立即处理？哪些记录为技术债务？"

[Step 2-4: 审视]
  AI: 引导开发者完成失误记录/AI协作评估/StartStopContinue
  开发者: 亲自填写（AI不能代写）

[AI自检] AI输出自检报告（审计是否逐类遍历/失误是否追溯到根因/新陷阱候选）

[Commit] 审视报告提交 + lessons/追加

[Feynman门禁] AI提问 → 开发者用自己的话回答
```
