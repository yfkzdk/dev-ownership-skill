# Dev-Ownership Skill — v0.2.0 改进路线图

> 基于 md2blog 项目实战验证 (2026-06-15) 发现的 9 个测量缺口。
> v0.1.0 解决了"流程框架"问题，v0.2.0 聚焦"质量可量化"问题。

---

## 缺口总览

```
当前 v0.1.0:  流程跑完了 → 能看出"做了什么"
             ↓ 缺第一层
v0.2.0-M1:   自动化门禁 → 能看出"做对了没有"
             ↓ 缺第二层
v0.2.0-M2:   行为量化 → 能看出"谁做的决策"
             ↓ 缺第三层
v0.2.0-M3:   效果验证 → 能看出"能力变了没有"
```

---

## M1: 自动化质量门禁（机器执行，不可跳过）

### G1.1 静态分析集成
- [ ] 集成 `ruff` 到 pre-commit hook（scripts/pre-commit-check.sh）
- [ ] 集成 `mypy --strict` 类型检查
- [ ] 集成 `bandit` 安全扫描
- [ ] 规约: **全部 zero-error 通过才允许 commit**（硬阻断）

### G1.2 测试质量测量
- [ ] 分支覆盖率: `pytest --cov --cov-branch --cov-fail-under=90`
- [ ] 变异测试: `mutmut run`，变异存活率 ≥70%（软警告）
- [ ] 规约: 覆盖率 <90% → 硬阻断；变异得分 <70% → 标注到 Review

### G1.3 性能基准
- [ ] 添加 `tests/benchmark/` 目录，测量 spec 中声明的性能指标
- [ ] CI 中每次 commit 跑 benchmark，回归 ≥20% → 告警
- [ ] 规约: benchmark 测试可选跳过（--skip-benchmark），但 Review 阶段必须运行

### G1.4 复杂度限制
- [ ] `radon cc -a src/` 圈复杂度，单函数 >10 → 审查标记
- [ ] `radon mi src/` 可维护性指数，<40 → 审查标记
- [ ] 规约: 软警告，记录到 Review 报告中

### 交付物
- `scripts/pre-commit-check.sh` 升级为完整门禁脚本
- `scripts/run-quality-gates.sh` 一键运行所有门禁
- `references/quality-gates.md` 质量门禁规范

## M2: 行为量化（数据驱动，纵向可比）

### G2.1 AI 依赖度追踪
- [ ] session log 新增字段:
  - `ai_lines_generated`: AI 生成的代码行数
  - `dev_lines_modified`: 开发者修改的代码行数（从 AI 输出中手动改的）
  - `ai_proposals_rejected`: 开发者拒绝 AI 方案次数
  - `dev_alternatives_requested`: 开发者主动要求备选方案次数
  - `ai_to_ai_review_chains`: AI→AI 审查链发生次数（消息路由检测）
- [ ] 项目结束时汇总: `ai_dependency_ratio = ai_lines_generated / (ai_lines_generated + dev_lines_modified)`
- [ ] 目标: 连续 3 个项目后 `ai_dependency_ratio` 下降趋势

### G2.2 时间审计
- [ ] session log 每条记录带时间戳
- [ ] 每阶段记录 `start_time` / `end_time`
- [ ] Review 阶段检查: "TDD 阶段时间 < Spec+Design 总时间？" → 警告（赶工信号）
- [ ] 规约: 时间分布异常 → 标记在 Retrospect 报告中

### G2.3 交互主动性评分
- [ ] session log 中的开发者输入分类:
  - 审批词（"可以"、"继续"、"OK"） → passive
  - 提问/追问/方案选择 → active
  - 纠正/拒绝/重新定义需求 → leading
- [ ] 项目结束计数器: `passive_count`, `active_count`, `leading_count`
- [ ] 目标: `leading` + `active` > `passive`

### G2.4 Spec 质量测量
- [ ] 追踪每个 spec 条款的生命周期:
  - `spec_clause_id`: 唯一标识
  - `created_in`: 哪个阶段创建
  - `revised_count`: 被修订次数
  - `bugs_from_ambiguity`: 因该条款歧义导致的 bug 数
  - `test_scenarios_planned`: 初始测试场景数
  - `test_scenarios_actual`: 实际实现的测试场景数
- [ ] 指标: `spec_revision_rate`（修订的条款数/总条款数）→ 高=spec 质量差
- [ ] 指标: `spec_leakage_rate`（TDD 阶段新增的测试场景/初始测试场景）→ 高=spec 遗漏多

### 交付物
- `templates/session-log-v2.md` — 新版 session log 模板（含量化字段）
- `scripts/analyze-session.py` — 自动统计 session log 量化指标
- `references/metrics-definition.md` — 指标定义与目标值

## M3: 效果验证（能力增长证据）

### G3.1 基线测量
- [ ] 项目启动前: 填写 `templates/pre-project-self-assessment.md`
  - 5-point Likert scale: "我是否经常跳过理解直接接受AI代码？"
  - 行为频率自评: "过去一周，AI审查AI的链发生过几次？"
- [ ] 项目结束后: 同一份量表复测
- [ ] 对比: 分数变化方向（↓ 在"消息路由"项上 = 改善）

### G3.2 Feynman 门禁客观化
- [ ] 录音转录（手动或工具辅助）
- [ ] 开发者事后自评打分（1-5），不是AI打分:
  - "我回答了为什么，还是只回答了是什么？"
  - "我引用了具体代码行号吗？"
  - "如果面试官追问，我能撑住第二轮吗？"
- [ ] 规约: 评分 <3 的阶段 → 标记为"未真正通过"，下次类似阶段需要改进方案

### G3.3 脱稿抽查
- [ ] Retrospect 阶段，AI 从代码中随机选 3 行
- [ ] 开发者脱稿解释每行的作用——录音或笔记
- [ ] 评分: 能解释（3/3）、部分解释（1-2/3）、无法解释（0/3）
- [ ] 目标: 连续 2 个项目后达到 3/3

### G3.4 陷阱复发检测
- [ ] 新项目开始时，AI 自动读取上次的 known-traps.md
- [ ] Spec/Design 阶段检查: 是否有陷阱的"预防规则"被违反？
- [ ] 如果有 → 硬中断，开发者必须确认"我知道这个陷阱，我选择忽略因为..."
- [ ] 指标: `trap_recurrence_rate`（复发的陷阱数/总陷阱数）→ 0% 是目标

### G3.5 审查有效性追踪
- [ ] Review 阶段标记的每个问题，追踪其生命周期:
  - `found_in_review`: 审查发现
  - `fixed`: 已修复
  - `was_real_bug`: 回顾判定——如果当时没修，会导致 bug 吗？
- [ ] 指标: `review_precision = was_real_bug / (was_real_bug + false_alarm)`
- [ ] 指标: `review_recall` — 交付后发现的 bug 中，有多少是 Review 应该发现但漏掉的？
- [ ] 但 `review_recall` 需要交付后追踪——所以 milestone 是 M3（至少需要 1 个项目交付后的数据）

### 交付物
- `templates/pre-project-self-assessment.md`
- `templates/feynman-self-score.md`
- `scripts/trap-check.sh` — 项目启动时自动检查 known-traps

## 优先级排序

| 优先级 | 缺口 | 原因 |
|:--:|------|------|
| P0 | G1.1 静态分析 | 机器执行，零成本，立即改善代码底线质量 |
| P0 | G1.2 测试质量 | 32个测试通过≠测试够好——mutation testing 是唯一客观的测试质量度量 |
| P1 | G2.1 AI 依赖度 | 这是 P0 问题 3.1 的直接测量——没有这个，skill 效果不可知 |
| P1 | G2.3 交互主动性 | 同上——"可以"/"继续"模式必须被量化才可能被修正 |
| P1 | G3.1 基线测量 | 没有基线就没有改善证据——skill 的价值无法被证明 |
| P2 | G2.4 Spec 质量 | 垃圾 spec → 垃圾项目——但需要 2-3 个项目数据才能建立基线 |
| P2 | G3.4 陷阱复发 | 需要多个项目积累陷阱库后才有效 |
| P3 | G1.3 性能基准 | 对 CLI 工具重要，对 Web API 项目可能不适用 |
| P3 | G3.5 审查有效性 | 需要交付后追踪，时间线长 |
