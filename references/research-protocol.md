# 搜索执行规范

> [来源: evanklem/evanflow brainstorming phase — "proposes 2-3 approaches with embedded grill"]
> [来源: 面试问题诊断书 §4.4 — 搜索vs学习的区别: 搜索是找一个答案解决眼前问题, 学习是内化思维模型解决所有类似问题]

## 何时搜索

Design阶段设计方案**之前**，必须先搜索。以下情况也必须搜索：

- 遇到不熟悉的领域概念
- 需要在多个技术方案之间做选择
- 声称"没有现成方案"之前（必须搜索验证过才能这么说）
- AI不确定某个API/库的行为时

## 搜索执行步骤

### Step 1: 确定搜索目标

AI明确写下要搜索什么：
- 核心问题: "我需要了解什么？"
- 关键词列表: 至少3组，包含同义词替换
- 搜索范围: GitHub / 技术文档 / 行业标准 / 学术论文

### Step 2: 执行搜索

- 最少搜索次数: 3次（不同关键词组合）
- 最少信息来源: 3个（不同域名的独立来源）
- 每次搜索记录到session log：搜索词 + 点击了哪些结果 + 为什么选择了这些结果

### Step 3: 整理搜索结果

AI输出搜索摘要：
```
## 搜索摘要
- 搜索词: [关键词1], [关键词2], [关键词3]
- 来源1: [URL] — [关键发现]
- 来源2: [URL] — [关键发现]
- 来源3: [URL] — [关键发现]

## 与当前需求的关联
- 可借鉴的模式: ...
- 不适用当前场景的模式（及原因）: ...
- 需要开发者注意的差异: ...
```

### Step 4: 搜索质量自检

AI自问：
- 搜索结果是否能支撑设计方案？
- 信息源是否多样（不只一个网站/组织）？
- 是否有遗漏的搜索角度？
- 搜索结果是否有时效性问题（超过2年的信息需标注）？

不通过 → 补充搜索。

## 禁止行为

- 禁止不搜索直接基于训练数据设计方案
- 禁止搜索一次就停止（如果第一次搜索结果不够充分）
- 禁止只搜GitHub不搜文档/标准
- 禁止使用来源不明的信息（必须标注URL）
