# 搜索执行规范

> [来源: evanklem/evanflow — brainstorming phase]
> [来源: 面试问题诊断书 §4.4 — 搜索vs学习的区别]
> [来源: mini-redis 复盘 — 三个参考源列出了名字但代码一行没读，RESP 解析靠已有知识写出来]

## 三通道分布式搜索

Design 阶段**之前**必须同时从三个通道搜索。**每个通道都必须有实质产出**——不能只列名字。

| 通道 | 搜索范围 | 实质产出要求 | 禁止 |
|------|---------|------------|------|
| **GitHub** | 开源项目、成熟实现、参考架构 | **clone 代码 + 读关键模块源码** | 只看 README 描述 |
| **Web/文档** | 技术规范、官方文档、行业标准 | **找到原始规范/标准文档** | 只看博客/教程 |
| **论文/学术** | ACM/IEEE/arXiv、行业白皮书 | **找到至少 1 篇相关论文/规范** | 以"找不到"为理由跳过 |

## 搜索深度自检

每个通道搜索完成后，AI 必须自问：

| 通道 | 自检问题 | 通过标准 |
|------|---------|---------|
| GitHub | 我读了参考实现的源码吗？我理解它的模块分解吗？ | 能说出参考实现的关键设计决策和代码结构 |
| Web | 我找到了官方规范/标准的原文吗？ | 能引用规范的具体章节号 |
| 论文 | 我找到了至少 1 篇吗？如果没有，我确认学术搜索引擎都没结果吗？ | 有论文引用或确认搜索了 Google Scholar/arXiv/ACM DL |

## 反模式——形式搜索

**这是当前 skill 最危险的失败模式**（已发生在 mini-redis 和 OpenSpec 发现中）：

```
形式搜索（无效）:
  搜索 "build your own Redis Python"
  → 列出 3 个 GitHub repo 名字
  → 没 clone、没读代码
  → RESP 解析器凭 AI 已有知识写
  → 声称"搜索了 3 个源"
```

**实质搜索（有效）**:
```
搜索 "RESP protocol specification"
  → 打开 redis.io/docs/latest/develop/reference/protocol-spec/
  → 读 Simple String/Error/Integer/Bulk String/Array 五种格式
  → 读 RESP3 的 null、double、boolean 等新类型

搜索 GitHub "Redis Python implementation"
  → clone 2 个参考实现
  → 读它们的 protocol.py——看别人怎么解析 RESP
  → 对比自己的实现——发现我漏了 RESP3 类型
```

## 搜索摘要模板

每次搜索完成，AI 输出：

```
## 三通道搜索摘要

### GitHub
- repo: [URL] — clone 并读取 [具体文件名]:[行号范围]
- 关键发现: [具体技术细节，不是模糊描述]
- 与当前设计的关联: [可借鉴的模式 / 不适用的部分及原因]

### Web/文档
- 文档: [URL] — 读取 [具体章节]
- 关键发现: [具体技术参数/规范要求]
- 与当前设计的关联: ...

### 论文
- 论文: [标题] or 确认搜索后未找到相关论文
- 关键发现: [如有]

## 对设计的直接影响
- 因为搜索结果，我采用了 [X] 模式
- 因为搜索结果，我拒绝了 [Y] 方案
- 因为搜索结果，我标记了 [Z] 为"已知但不在范围内"
```

## 禁止行为

- 禁止不搜索直接基于训练数据设计方案（mini-redis 的 RESP 解析器就是基于训练数据写的）
- 禁止搜索了但只列名字不读代码（形式搜索）
- 禁止跳过论文通道（即使找不到相关论文，也要确认搜索过了）
- 禁止使用来源不明的信息（必须标注 URL）
- 禁止搜索一次就停止（如果第一个通道不充分，补充搜索）

## 与 Spec Step 0b 的关系

Spec 阶段的 Step 0b（对标研究）使用的是这个搜索协议。Step 0b 的"对标研究报告"**必须包含三通道搜索摘要**——不能只是功能列表对比。
