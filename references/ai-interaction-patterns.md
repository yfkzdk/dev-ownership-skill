# AI 交互模式表

> [来源: Anthropic 2025 RCT — Shen & Tamkin, N=52, 6种交互模式]
> [来源: Sankaranarayanan 2026 — "Epistemic Debt" 实验, N=78]

## 核心原则

**不是所有AI使用方式都有害——关键在于你怎么用。**
"生成后理解"模式的得分（86%）甚至超过了不用AI的对照组（67%）。

## 六种模式

### 保持/增强学习的模式

| 模式 | 学习效果 | 何时使用 | 示例提示词 |
|------|---------|---------|-----------|
| **生成后理解** | **增强** (测验分 86%) | 主要开发模式 | "生成代码后，逐行解释每行的作用。我会追问我不理解的。" |
| **混合代码-解释** | 保持 (测验分 68%) | 正常开发节奏 | "实现这个功能，同时在关键决策点解释你为什么这样设计。" |
| **概念探究** | 保持 (测验分 65%) | 遇到不熟悉领域时 | "先不要写代码。解释一下Trie数据结构如何用于最长前缀匹配，以及和sorted list方案的对比。" |

### 损害学习的模式

| 模式 | 学习效果 | 风险 | 何时允许 |
|------|---------|------|---------|
| **AI全权委托** | 损害 (测验分 39%) | 开发者不了解代码细节 | 仅限纯样板代码（.gitignore、格式化配置等） |
| **渐进式依赖** | 损害 (测验分 35%) | 开始手动，遇到困难就全部交给AI | **禁止**——遇到困难正是学习机会，应切换到概念探究模式 |
| **迭代式AI调试** | **严重损害** (测验分 24%) | 错误信息丢给AI，自己不看stack trace | **禁止**——必须先自己读错误栈，尝试定位，然后才找AI辅助 |

## 模式选择决策树

```
接到开发任务
    │
    熟悉这个领域吗？
   ┌────┴────┐
  是          否
   │           │
   │      切换到「概念探究」模式
   │      先理解概念，再开始开发
   │           │
   │      理解后 ↓
   │          
使用「混合代码-解释」或「生成后理解」模式
   │
   遇到困难/错误
   │
   先自己读错误栈、尝试定位（至少5分钟）
   │
   仍然无法解决？
   │
   切换到「概念探究」模式
   问AI："这个错误的原因是什么？我自己分析的是X，但不确定。请帮我验证。"
   │
   理解原因后自己修复
```

## 禁止模式的使用场景

- 当你发现自己在说"帮我修一下这个bug"而自己没有看过错误信息 → 打断自己，先读错误栈
- 当你发现AI写了代码你直接Accept而没有逐行理解 → 回退，要求AI附带逐行解释
- 当你发现自己在用另一个AI审查第一个AI的代码 → 停止。先用自己的5轴清单过一遍。你的审查能力必须独立于AI工具链
